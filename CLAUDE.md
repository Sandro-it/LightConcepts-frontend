# LightConcepts / Svitli Idei — контекст проєкту

## Що це за проєкт

Інтернет-магазин крафтових світильників, свічок та меблів у стилі лофт
("Svitli Idei" / LightConcepts), який я (Саня) писав самостійно під час
навчання (ідея з часів ковіду). Проєкт не завершений — тепер добудовуємо
разом з Claude.

Це не суто e-commerce, а радше **різнопланова візитна картка**: каталог
товарів + сторінка "Незламність" (`/resilience`) — благодійна сторінка про
військові ліхтарі, які я сам виготовляю і відправляю на фронт (тема:
війна росії проти України). Може згодом додатись контент про дизайн тощо.

Мультимовність обов'язкова: **українська, англійська, німецька** (i18next).

## Технічний стек

- React 18 + Vite 5, React Router v6, i18next, axios, CSS Modules
- Бекенд: **Strapi CMS** (потрібно піднімати заново, дані втрачені)
- Репозиторій: `Sandro-it/LightConcepts-frontend` на GitHub
- Раніше: Hetzner VPS (бекенд) + Vercel (фронтенд) — обидва зараз офлайн
- Домен: `svitli.com.ua` / `api.svitli.com.ua`

## Історія проблеми (чому починаємо частково з нуля)

Оригінальний Strapi-бекенд на Hetzner VPS вимкнули через несплату.
База даних товарів, ймовірно, втрачена (без бекапів). Паролі до Hetzner і
Strapi забуті — спроба відновити доступ ведеться паралельно, але
розробку не блокуємо очікуванням.

**Рішення: рефакторити фронтенд, а не переписувати з нуля.** Код
адекватний (React 18 + Vite, не застарілий), переклади й CSS вже готові —
шкода викидати. Бекенд і так піднімається заново (дані втрачені), тому
там свобода дій.

## Хостинг — рішення ще не прийняте

Обговорювані варіанти (переваги/недоліки з'ясовані, фінального вибору
ще нема):
- **Відновити Hetzner** — найменше нової роботи, ~€4-5/міс, треба
  автоплатіж + автобекапи цього разу (щоб не повторити втрату даних)
- **Strapi Cloud Free Plan** — офіційно безкоштовний назавжди, без картки,
  але є розбіжності в джерелах щодо лімітів і чи дозволене комерційне
  використання — перевірити на офіційній сторінці перед вибором
- **Уникати Render.com** — безкоштовна Postgres-база видаляється через
  30-90 днів, той самий ризик втрати даних, що і був
- **Для фото товарів окремо**: розглянути Cloudinary (безкоштовний план,
  плагін для Strapi) — розв'язує зберігання фото незалежно від вибору
  хостингу бекенду

## Git-workflow

Працюємо гілками: `feature/...`, `fix/...`, `chore/...` → коміт → мердж →
теги релізів, аналогічно до попереднього проєкту з фотками. Це не
"додаток", а сайт, тому мітки радше по завершенню фаз/релізів, ніж по
кожній дрібниці.

## Принципи, які варто пам'ятати

- Рефакторинг, не переписування — код вартий збереження
- DRY з самого початку в новому коді (уникнути повторення 7 категорій)
- Автоматичні бекапи і документовані доступи — обов'язково, це і була
  причина втрати даних минулого разу
- Проєкт некомерційно-масштабний (крафтові вироби, не розкручений
  магазин) — рішення з хостингу/інфраструктури мають враховувати саме
  такий масштаб, а не enterprise-рівень

## Правило DRY — явно, бо між сесіями контекст не переноситься

7 сторінок категорій (див. вище) з'явились через copy-paste "на швидку
руку" — і саме так народився баг з переплутаним кольором. Тому:

- **Правило трьох повторень**: перший раз — пиши прямо. Другий раз можна
  ще скопіювати. На **третьому** схожому файлі/компоненті — зупинись і
  запитай, чи не варто винести спільну логіку, а не мовчки копіювати далі
- Це стосується не лише React-компонентів, а й схеми Strapi (використовуй
  **Components** для повторюваних груп полів, наприклад SEO-полів чи
  специфікацій товару, а не дублюй ті самі поля в кожному content-type)
- Конкретні місця ризику в цьому проєкті:
  - Фаза 3 (checkout): форми доставки/оплати — не копіювати `<input>`-и,
    винести спільний `AddressForm`
  - Фаза 4 (`/contact`, `/shipping`, `/orders`, `/favorites`, `/account`):
    швидше за все, знадобиться спільний `PageLayout` (заголовок + кнопка
    "назад" + обгортка), той самий патерн, що вже є в `CategoryPage.jsx`
- Не абстрагувати завчасно "про всяк випадок" — це так само шкідливо, як
  copy-paste (over-engineering). DRY застосовується постфактум, коли
  повторення вже видно, а не наперед
